Internal Spec · Phase 2 Initial Build

Graduated Dissent Benchmark
Data Pipeline Tool

A single tool that handles every stage of the benchmark pipeline from candidate extraction through inter-rater scoring, architected for two people now and contributors later.

Andrew Brilliant  ·  Joe Reid
Phase 2 Local deployment Target n = 25–50

Goals & Non-Goals

What this tool does

Grows the benchmark from n=10 to n=50 as a matter of steady workflow execution rather than ad-hoc scripting. Enforces a locked schema so every contribution has consistent structure. Creates an audit trail for every decision that can be reconstructed later. Designed for two users now, architected so adding contributors in phase 3 does not require a rewrite.

What this tool does not do

  • Not a real-time collaboration platform — one user at a time is fine for phase 2
  • Not a general retraction database — only tracks what is in the benchmark
  • Not a model serving platform — calls external APIs, does not host models
  • Not a replacement for git — git still holds the canonical dataset and history
  • Never stores API keys on the server, under any circumstances

End-to-end pipeline with human, mixed, and automated stages
The meta-rule for the tool: anywhere the temptation is to use an LLM because it would be faster, ask whether a wrong answer would be detectable. If yes, automate with LLM assistance. If no, humans only. The default bias should be humans, with LLMs used where their output is self-verifying.

Users & Roles

Phase 2 roles

  • Maintainer (Drew) — full access, can lock schemas and ground truth, can override any automated decision
  • Contributor (Joe) — full access within assigned domains, can propose changes to locked items for maintainer approval

Added in phase 3

Data Model

Every entity has standard audit fields: created_at, created_by, updated_at, updated_by. The schema is locked at the start of phase 2 so all contributions share the same structure.

Paper
record_id · retraction_watch_id · title_hash · pub_date · retraction_date · reason_raw · reason_structured · class (A/B/C) · status · exclusion_reason
RetractionNotice
paper_id · notice_text · notice_url · parsed_fields (JSON)
Control
control_id · paired_paper_id · journal · topic · pub_date · verification_notes
GroundTruthKeywords
paper_id · keyword_groups (JSON) · locked (bool) · locked_by · locked_at
AnonymizedManuscript
paper_id · file_path · anonymization_log
ProtocolRun
run_id · paper_id · config_snapshot · model_versions · timestamp
Finding
run_id · stage · severity · text · source_model
ScoringResult
run_id · paper_id · gt_match · matched_findings · overrides
Contributor
user_id · name · domains · papers_added · inter_rater_subset

Stage Detail

Each stage describes its trigger, what the system does, what the user sees and does, and what gets stored.

STAGE 01
Candidate Ingestion from Retraction Watch
Automated
Trigger
Maintainer clicks "Update candidates"
System
Pulls latest CSV from Crossref mirror via git pull, diffs against existing records, applies pre-filter (post-cutoff, excludes misconduct categories, excludes paper mills)
User sees
Table of new candidates with filters for date, journal, and reason category
User does
Selects candidates to add to the pool or accepts the pre-filter default
Stored
Paper records with status=candidate, linked to Retraction Watch ID
STAGE 02
Notice Enrichment
LLM + Verify
Trigger
Automatic after new candidates are added
System
Fetches retraction notice text and full-text availability via Unpaywall API; LLM parses notice into structured fields
User sees
List of parsed results with confidence markers; low-confidence parses highlighted
User does
Spot-checks a sample (default 20%, minimum 5); approves, edits, or flags
Stored
RetractionNotice records with parsed_fields and audit trail of verified entries
STAGE 03
Triage (Class A / B / C)
Human
Trigger
User opens triage queue
System
Shows unreviewed candidates with notice text, LLM-suggested class, and pre-specified criteria reminder
User does
Reads notice, assigns class, records decision reasoning as free text
Stored
Class decision, reasoning, LLM suggestion (kept for drift analysis), human-LLM agreement flag
STAGE 04
Domain Routing & Expert Review
Human Expert
Trigger
Triage assigns a paper to a domain outside the maintainer's expertise
System
Routes to contributor whose declared domains match, or to queue if no match
Contributor sees
Notice, abstract, methodology section, relevant figures
Contributor does
Confirms the error is text-detectable and matches the notice, or rejects with reason
Stored
Verification record with contributor ID and reasoning
STAGE 05
Anonymization
Script + LLM
Trigger
Paper reaches accepted status
System
Runs anonymization script (strips author names, affiliations, journal, dates, acknowledgments, references); LLM scans for remaining identifying markers
User sees
Before/after diff plus LLM-flagged items
User does
Approves each flag or marks as false positive
Stored
AnonymizedManuscript file plus full anonymization log
STAGE 06
Control Matching
LLM + Verify
Trigger
User opens control matching interface for a paper
System
Suggests candidate controls from same journal, similar date, topic overlap; LLM ranks by similarity
User does
Reviews top 5 candidates, selects a match or rejects all and requests new suggestions
Stored
Control record paired to retracted paper with verification notes
STAGE 07
Ground Truth Keyword Extraction
Human Lock
Trigger
User opens ground truth editor for a paper
System
Shows notice text and current keyword groups (may be empty or LLM-suggested)
User does
Edits groups, adds, removes, then clicks "Lock" when satisfied
Stored
Keyword groups as JSON, lock status, lock timestamp; post-lock changes require explicit unlock with audit reason
STAGE 08
Protocol Execution
Automated
Trigger
User selects paper(s) and clicks "Run protocol"
System
Prompts for API keys (session-scoped, never stored); runs provers, judge, steelman, arbiter; saves all outputs
User sees
Progress indicator with cancel option, final output with all stage artifacts
Stored
ProtocolRun record, Finding records, raw output files
STAGE 09
Automated Scoring
Auto + Audit
Trigger
Automatic after protocol run
System
Runs keyword matching against locked ground truth; produces audit table
User sees
Paper-by-paper matches with finding text and which keyword group matched or did not
User does
Reviews audit, optionally overrides with reason
Stored
ScoringResult record and complete override log
STAGE 10
Inter-rater Check
Human Only
Trigger
Second user opens an already-scored paper
System
Hides first user's decisions, shows only findings
Second user does
Scores independently
System then
Compares, computes agreement metric, surfaces discrepancies for discussion
Stored
Second rater decisions, agreement stats, discrepancy log

Technical Architecture

Stack

  • Backend — Python with FastAPI, single process, SQLite database file; SQL with a thin helper layer rather than a full ORM (readable for someone learning Python)
  • Frontend — Server-rendered HTML with htmx for interactivity; no build step, no JavaScript framework
  • File storage — Local filesystem, organized by paper record_id, directory layout mirrored in git so the full dataset is versioned
  • LLM integration — Single interface wrapping provider APIs; keys provided at runtime and never logged, stored, or sent to the backend beyond the scope of the request
  • Auth — None in phase 2 (single-user local); password-hashed accounts in phase 3; OAuth optional for phase 4
  • Deployment — python -m benchmark_tool locally or inside a Docker container; SQLite file committed to repo each session; hosted on a modest VPS in phase 3

Phase Boundaries

PHASE 2
Pilot Tool
Drew & Joe, local deployment. Grow benchmark to n=25–50 with the tool handling the pipeline. Prove the workflow works before scaling.
PHASE 3
Small Group
5–10 contributors, hosted deployment, password auth, review queue for external submissions. Same data model, bigger user base.
PHASE 4
Open Contribution
Open contribution model with rate limiting, moderation, and attribution on each release. Conditional on phase 3 traction.

Build Order

Minimum viable for phase 2

  1. Stage 01 (candidate ingestion from CSV) — highest leverage because it unlocks the whole pipeline and replaces the LLM-guessing selection method that produced only 10 papers last time
  2. Stage 03 (triage UI) — where human judgment gets applied consistently
  3. Stage 07 (ground truth editor) — must exist before scoring can be trusted
  4. Stage 09 (automated scoring integration) — already mostly works as a script, needs to live in the tool so the audit table is first-class data

Stages 02, 05, 06, 08, and 10 can remain scripted or manual in the short term and get wrapped into the tool as they become bottlenecks.

Suggested division of labor

  • Drew — database schema, Stage 09 integration, overall architecture decisions, spec enforcement
  • Joe — Stages 01, 03, and 07 user interfaces (SQL, FastAPI basics, git integration, API clients, simple prompt engineering, htmx); gives real ownership of meaningful pieces rather than fetch-tasks, matches the AI/IT pivot
  • Both — Stage 04 workflow because domain routing affects everyone

Open Questions

Things to decide Friday rather than tonight.

  • Database migration path — SQLite is right for phase 2, but the phase 3 Postgres path should be checked early so SQL stays portable
  • Git auto-commit — should the tool auto-commit changes to the dataset, or should the user commit manually? Auto-commit is safer but noisier
  • LLM provider abstraction — support one provider initially or multiple from the start? Multiple adds complexity but matches the multi-model philosophy of the paper
  • Anonymization automation boundary — how much can be scripted reliably? Requires a small test set to evaluate
